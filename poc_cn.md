### **灵泉Flow (WorshipFlow) App**

**版本:** 0.1
**日期:** 2023年10月27日
**目标:** 验证使用LLM（大型语言模型）根据证道主题和诗歌上下文，生成高质量、多维度敬拜串词的核心可行性。

---

### **1. 概述与指导原则**

* **核心目标:** 快速搭建一个可用原型，让敬拜主领可以输入证道主题、选择诗歌，并生成可用的串词草稿。
* **技术哲学:**
  * **极简主义:** 只实现最核心的功能，忽略所有非必要功能（如用户登录、云同步、团队协作等）。
  * **快速迭代:** 使用高生产力工具，以便快速修改和演示。
  * **本地优先:** 所有数据存储在本地，不涉及数据库和复杂的后端服务。

---

### **2. 系统架构 (KISS)**

整个POC是一个单体应用，架构极其简单：

```
+----------------+      +-------------------------+      +-----------------+
|   用户 (User)  | <--> |  Streamlit Frontend/UI  | <--> | Python Backend  |
+----------------+      |   (app.py)              |      | Logic (in app.py) |
                        +-------------------------+      +--------+--------+
                                                                  |
                                                                  |
                                            +---------------------+---------------------+
                                            |                                           |
                                   +--------v--------+                         +--------v--------+
                                   |   Local File    |                         |   LLM API       |
                                   |   Storage       |                         |   (e.g., Gemini 2.5 Pro)|
                                   | (JSON/Markdown) |                         +-----------------+
                                   +-----------------+

```

* **前端/UI:** 使用 `Streamlit` 库，它能用纯Python脚本快速生成Web UI。
* **后端逻辑:** 全部写在 `app.py` 脚本中，负责处理UI交互、文件读写和API调用。
* **数据存储:** 使用本地文件夹和 `JSON` 文件来管理诗歌和敬拜流。
* **AI核心:** 通过API调用外部的大型语言模型服务。

### **4. 数据模型与存储**

为了简单起见，我们使用 `JSON` 格式，因为它比Markdown更易于结构化解析。

#### **4.1 诗歌数据模型 (`song.json`)**

每首诗歌存储为一个单独的JSON文件，放置在 `data/songs/` 目录下。

* **文件名示例:** `data/songs/here_i_am_to_worship.json`
* **文件内容结构:**

```json
{
  "title": "Here I Am to Worship",
  "author": "Tim Hughes",
  "key": "D",
  "lyrics": "Light of the world\nYou stepped down into darkness\nOpened my eyes, let me see\n...",
  "tags": ["赞美", "敬拜", "道成肉身", "安静"]
}
```

#### **4.2 敬拜流数据模型 (`flow.json`)**

每次设计的敬拜流程也保存为一个JSON文件，放置在 `data/flows/` 目录下。

* **文件名示例:** `data/flows/2023-10-29_sunday_service.json`
* **文件内容结构:**

```json
{
  "date": "2023-10-29",
  "sermon_title": "行在光明中",
  "key_scripture": "约翰一书 1:7",
  "worship_flow": [
    {
      "type": "song",
      "song_id": "here_i_am_to_worship",
      "generated_intro": {
        "praise": "弟兄姊妹，让我们以一颗敬畏的心开始今天的敬拜。约翰福音说'光来到世间'，就如这首歌所唱，祂是世界的光，为我们走进黑暗...",
        "exhortation": "..."
      }
    },
    {
      "type": "transition_text",
      "content": "（这里是连接上一首和下一首歌的串词）"
    },
    {
      "type": "song",
      "song_id": "how_great_is_our_god",
      "generated_intro": {}
    },
    {
        "type": "prayer",
        "generated_prayer": "主啊，我们感谢你..."
    }
  ]
}
```

*`song_id` 对应诗歌JSON文件的文件名（不含扩展名）。*

### **5. 核心模块POC实现方案 (Streamlit)**

主应用 `app.py` 将使用 `st.sidebar` 来导航不同的功能页面。

#### **模块一：诗歌库管理 (灵感库)**

* **UI:** 在侧边栏创建一个 "诗歌库管理" 页面。
* **功能:**
  1. **添加新歌:** 使用 `st.text_input` 录入标题、作者，`st.text_area` 录入歌词，`st.multiselect` 选择预设的标签。点击 `st.button("保存诗歌")` 后，程序将这些信息整合成一个JSON文件并存入 `data/songs/` 目录。
  2. **查看/搜索诗歌:** 使用 `st.selectbox` 或 `st.text_input` 作为搜索框，列出 `data/songs/` 目录下的所有诗歌标题。选中后，在主页面展示该诗歌的完整信息。

#### **模块二：敬拜流设计器**

* **UI:** 主页面。
* **功能:**
  1. **输入主日信息:** 顶部用 `st.text_input` 让用户输入“证道主题”和“核心经文”。
  2. **选择诗歌:** 使用 `st.multiselect` 列出所有可用诗歌，用户按敬拜顺序进行选择。
  3. **展示流程:** 将用户选择的诗歌顺序显示出来。在每两首歌之间，放置一个 `st.button("生成串词")`。

#### **模块三：串词生成器 (Weave Engine)**

这是POC的核心。

* **UI:** 集成在“敬拜流设计器”中。
* **功能:**
  1. **触发:** 当用户点击两首歌之间的 `st.button("生成串词")`。
  2. **后端逻辑:**
     a. 读取 **证道主题** 、 **核心经文** 。
     b. 读取**上一首歌**的歌词和主题标签。
     c. 读取**下一首歌**的歌词和主题标签。
     d. **构建Prompt:** 这是关键。将以上信息整合成一个结构化的Prompt，发送给LLM API。

     * **Prompt示例:**
       ```
       你是一位非常有经验的、属灵的基督徒敬拜主领。请根据以下信息，为两首敬拜诗歌之间撰写一段连接性的串词。
       ```
     * 主日证道主题：'行在光明中'
     * 核心经文：'约翰一书 1:7'
     * 上一首歌：《Here I Am to Worship》，歌词大意是关于耶稣是世界的光，我们敬拜祂。
     * 下一首歌：《How Great Is Our God》，歌词大意是宣告神的伟大和威严。

     ```
      请提供三个不同维度的串词草稿：
      1. 【赞美维度】: 侧重于引导会众从对耶稣的个人敬拜转向对神永恒属性的集体宣告。
      2. 【激励维度】: 侧重于鼓励会众，即使在黑暗中行走，也能因着神的伟大而充满盼望。
      3. 【祷告格式】: 直接写一段可以带领会众开口的祷告。
     ```

     ```

     e. **调用API:** 使用 `openai.ChatCompletion.create()` 发送请求。
     f. **展示结果:** 将LLM返回的多个维度串词，使用 `st.expander` 或 `st.tabs` 展示在按钮下方，并提供一个 `st.text_area` 让用户可以复制和修改。

     ```

#### **模块四：操练室 (简版)**

* **UI:** 一个独立的页面 "敬拜排练"。
* **功能:**
  1. 用户在“敬拜流设计器”中完成设计后，点击 `st.button("进入排练模式")`。
  2. 应用将整个流程（诗歌标题、歌词、用户最终确定的串词/祷告）整合成一个大的Markdown字符串。
  3. 在新页面使用 `st.markdown()` 将其完整、清晰地展示出来，作为一个简单的“电子版讲稿”或“提词器”。

### **7. POC开发步骤**

1. **环境搭建:** `pip install streamlit`。设置Gemini service account 或者api-key。
2. **构建数据层:** 创建 `data/songs` 目录，并手动创建2-3个 `song.json`文件用于测试。
3. **开发模块一:** 实现诗歌的添加和查看功能。
4. **开发模块二:** 实现敬拜流的设计界面（主题输入、诗歌选择）。
5. **核心开发 - 模块三:** 编写Prompt构建逻辑，集成Gemini 2.5 Pro API，并展示生成结果。这是最关键的一步。
6. **开发模块四:** 实现简单的排练展示页面。
7. **测试与反馈:** 邀请1-2位敬拜主领试用，收集关于串词质量和可用性的反馈。
